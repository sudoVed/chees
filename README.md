# Chess — a from-scratch browser chess engine (C++ → WebAssembly) with a neural-net brain

A complete chess game that runs entirely in your browser. The rules **and** the
AI are hand-written in C++, compiled to WebAssembly (WASM), and driven by a tiny
vanilla-JavaScript front end. No Stockfish, no chess.js, no framework, no
backend server. The neural network that evaluates positions (an **NNUE**) was
trained from scratch.

This README is written to be read by someone who has never seen the project (or
much chess-engine code) before. If a term shows up, it gets explained.

## 1. What it is, in one minute

- **A real chess engine.** Legal moves, checks, checkmate, stalemate, the
  threefold-repetition / 50-move / insufficient-material draws, castling, en
  passant, promotion — all generated by the C++ engine. The UI never decides
  what's legal; it only draws the moves the engine reports. That's the core
  design rule: **the engine is the single source of truth.**
- **Two brains you can pick between.** A *hand-crafted* evaluator (HCE — classic
  chess heuristics written by a human) and a *neural network* evaluator (NNUE — a
  small net trained on millions of positions). You choose which one the AI uses,
  and at what search depth.
- **An opening book.** For known opening positions the AI plays a real book move
  instead of thinking.
- **Fast.** All the heavy work is WASM (near-native speed), it uses CPU vector
  instructions (SIMD), and the search runs on a background thread so the board
  never freezes.
- **Polished to play.** Piece-slide animation, move/capture/check/castle/promote
  sound effects, optional background music, captured-piece ("trophy") trays, a
  board-view toggle for two players sharing a phone, and a deliberate short pause
  before the AI moves so it doesn't snap.
- **Static.** The whole thing is just files. It hosts on GitHub Pages or
  Cloudflare Pages with no server.

## 2. Run it / deploy it

### Run locally

You need to serve the `frontend/` folder over HTTP. Don't open `index.html` as a
`file://` — browsers block the background worker and the network fetch under
`file://`.

```bash
cd frontend
python -m http.server 8000        # then open http://localhost:8000
#   or:  npx serve frontend
```

> **Tip:** if you change files and the browser still shows the old version, that's
> the browser cache, not the code. Hard-reload (Ctrl/Cmd+Shift+R) or use a private
> window. `frontend/serve.json` already tells `npx serve` not to cache.

### Deploy (no backend)

- **Cloudflare Pages:** connect the repo, Build command = *(none)*, Output
  directory = `frontend`.
- **GitHub Pages:** push the repo, Settings → Pages → Source = **GitHub Actions**
  (`.github/workflows/pages.yml` publishes `frontend/`).

Everything the site needs — the engine, the neural net (`frontend/models/net.nnue`),
and the opening book (`frontend/book/Book.bin`) — lives inside `frontend/` and is
fetched as plain files.

## 3. How the whole thing fits together

```
┌──────────────────────── Browser ────────────────────────┐
│                                                          │
│  Vanilla JS front end (the only thing that "paints")     │
│   • draws the board + pieces (inline SVG)                │
│   • captures clicks, shows legal-move dots               │
│   • menus: new game / undo / exit / pick AI              │
│        │  asks "what's legal?", "make this move"         │
│        ▼                                                  │
│  ┌───────────────── WASM engine (C++) ───────────────┐   │
│  │  bitboards · move generation · rules              │   │
│  │  evaluation (HCE or NNUE) · alpha-beta search     │   │
│  └───────────────────────────────────────────────────┘   │
│        ▲                                                  │
│        │  the AI search runs in a Web Worker (separate    │
│        │  thread) so the UI never freezes while it thinks │
└──────────────────────────────────────────────────────────┘
        +  Opening book probed in JS (Book.bin)
```

**Why C++ → WASM?** Chess engines do *enormous* amounts of tiny operations
(millions of positions per move). JavaScript is too slow for that. WebAssembly
runs at near-native speed. So all the chess logic lives in C++ and is compiled to
a `.wasm` module; JavaScript only handles the screen.

**One source of truth.** The same legal-move generator feeds both the on-screen
highlights and the AI search. The UI literally cannot let you make an illegal
move, because it only offers the squares the engine returned.

**How the WASM gets loaded.** The compiled engine is embedded as a base64 string
inside `frontend/js/engine-wasm.js`. That means there's no separate `.wasm` file
to fetch and no CORS headaches — the page and the worker both just decode the
string and instantiate it.

## 4. Playing: modes, opponents, the opening book

From the start screen you pick **Human vs Human** or **Play with AI**.

For Play-with-AI you choose a **model** and which colour you play. A "model" is
just a named combination of *which evaluator* and *how deep it searches*:

- **NNUE models** think with the neural network.
- **HCE models** think with the hand-crafted evaluator.
- The two never mix. An NNUE model uses *only* the net; an HCE model uses *only*
  the HCE. (If you pick an NNUE model and the net can't load, the game refuses to
  start rather than secretly playing HCE.)

All the models are defined in **one place** — the `MODELS` object at the top of
[`frontend/js/controls.js`](frontend/js/controls.js). Each entry has:

| field    | meaning                                                            |
|----------|-------------------------------------------------------------------|
| `label`  | the text shown in the dropdown                                    |
| `eval`   | `'nnue'` or `'hce'` — which brain                                  |
| `depth`  | how many plies deep the search goes                               |
| `timeMs` | time budget per move (ms); it returns the best move from the last finished depth if it would overshoot |
| `desc`   | one-line hint shown under the dropdown                            |

The dropdown is generated from this object, so to add/remove/rename opponents you
just edit the object — nothing else. (See [§13](#13-tweaking-things).) The shipped
set is **eight** opponents — NNUE and HCE each at depths 3, 5, 10, and a depth-20
preset that is really "think about 5 seconds" (the time budget stops the search
long before depth 20). They have playful names (`Neurally Oblivious`,
`Classically Heroic`, …); the real spec shows under the dropdown.

### The move readout (top bar)

- **Human vs Human:** the side-to-move indicator (a dot + "White to move").
- **Vs AI:** after the AI replies, the bar shows its move in proper algebraic
  notation in gold (`Qh4`, `O-O`, `exd5`, `e8=Q+`, `Qh7#` — with full
  disambiguation), followed by a smaller grey tag, `via NNUE D10 · 300 ms`, that
  reports the eval that **actually** ran, the depth reached, and the time taken
  (or `Book move` while in book).

### Captured pieces (trophy trays)

Two trays flank the board showing each side's **captures** (the opponent pieces it
has taken), chess.com-style, ordered Q, R, B, N, P. On wide screens they sit
vertically left and right of the board; when width is tight they stack above and
below. Captures are tracked from the move stream (not a board diff), so
**promotions are handled correctly**: a pawn that promotes is never shown as a
captured pawn, and a captured promoted queen correctly shows as a dead queen.
Undo rolls the trays back in step.

### Board view (Human vs Human)

By default the whole board flips each turn to face the side to move. A toggle in
the top bar (a board-rotation / crossed-out icon) switches to **table mode**: the
board stays fixed and instead **all the pieces rotate 180° on Black's turn**, so
two people sitting at opposite ends of a phone each read the board from their
side. (The choice is remembered.)

### Sound & music

Two icon toggles — a speaker (sound effects) and a music note (background music) —
appear on both the home screen and in-game, and the preference is remembered.
Effects map to the move type (move / capture / castle / promotion / check), plus a
game-end chime and button hovers. Undo plays the move sound too.

### Other touches

The AI waits at least half a second before playing (instant moves look jarring),
Exit fully stops the game and returns home, and the game-over banner has an **×**
so you can dismiss it and study the final position.

## 5. The brain, part 1 — the search

When it's the AI's turn (and it's out of book), it **searches**: it looks ahead,
trying moves, replies, replies-to-replies, and so on, scoring the leaf positions
with an evaluator and backing the scores up.

The core is **minimax with alpha-beta pruning**. Minimax = "I pick the move that's
best for me assuming you'll reply with what's best for you." Alpha-beta = a way to
*skip* branches that can't possibly change the decision, so you search far fewer
positions for the same answer.

On top of that, the engine uses every standard speed/strength trick (all in
[`engine/search/search.cpp`](engine/search/search.cpp)):

- **Iterative deepening** — search depth 1, then 2, then 3… Sounds wasteful, but
  the results of the shallow searches make the deeper ones prune *much* better,
  and it lets us stop anytime and still have a good move (handy with a time
  budget).
- **Transposition table (TT)** — a big hash table that remembers positions
  already scored (chess often reaches the same position by different move orders).
  Keyed by a **Zobrist hash** of the position.
- **Move ordering** — alpha-beta prunes best when you try good moves first. The
  engine orders moves by: the TT's best move, then captures (**MVV-LVA**: grab the
  most valuable victim with the least valuable attacker), then "killer moves" and
  a "history" table of quiet moves that worked elsewhere.
- **Quiescence search** — at the bottom of the search, before scoring, keep
  resolving captures so you don't stop in the middle of a trade and get a silly
  score.
- **Null-move pruning** — "if I pass my turn and I'm *still* winning, this line is
  too good for the opponent; prune it." Cheap and powerful.
- **Late-move reductions (LMR)** — moves tried late in the list are probably bad,
  so search them shallower; only re-search at full depth if one surprises us.
- **Aspiration windows** — start each new depth expecting a score near the last
  one, with a narrow search window; widen only if reality falls outside it.
- **Contempt (avoid draws)** — a draw is scored as slightly bad (~0.3 of a pawn)
  for the side to move, so the AI keeps fighting in roughly equal positions
  instead of taking lazy repetition / 50-move draws. It still accepts a draw when
  the alternative is clearly worse.

The search talks to the evaluator through a small interface (`Evaluator`), so the
*same* search code runs with either the HCE or the NNUE. Swapping brains changes
nothing about the search.

## 6. The brain, part 2 — evaluation (HCE and NNUE)

The search needs a number for "how good is this position." That number comes from
an **evaluator**.

### HCE — the hand-crafted evaluator

[`engine/eval/hce.cpp`](engine/eval/hce.cpp). This is classic, human-written chess
knowledge turned into a score: material (a queen is worth ~9 pawns), piece-square
tables (a knight in the centre is better than on the rim), king safety, pawn
structure (doubled/isolated/passed pawns), rook on open files, mobility, and so
on — all blended and tapered between middlegame and endgame. It's fast and it
plays a decent game. No training required; a human encoded the ideas.

### NNUE — the neural-network evaluator

NNUE stands for "**E**fficiently **U**pdatable **N**eural **N**etwork" (the
letters are famously backwards). Instead of a human writing the rules, a small
neural network *learns* to score positions from millions of examples.

"Efficiently updatable" is the whole point and is explained in
[§7](#7-the-efficiency-tricks-explained-this-is-the-good-part). First, the shape
of the net (in [`engine/nn/nnue.cpp`](engine/nn/nnue.cpp), trained by the code in
`training/`):

```
   40960 inputs  ─►  256 (per side, ×2 = 512)  ─►  32  ─►  32  ─►  1 score
   (HalfKP features)   "accumulator"            two hidden layers   centipawns
```

**The inputs (HalfKP).** This is the clever input encoding NNUE nets use. For
each side's perspective, every feature answers: *"my king is on square K, and
there's a (piece-type P) on square S."* With 64 king squares × 10 piece types (5
of mine + 5 of yours, kings excluded) × 64 squares = **40,960** possible features.
In any given position only ~30 of them are "on." Tying every feature to the king's
location is what makes the net understand king safety so well — and it's also why
the efficiency trick has a special case for king moves (see below).

**The layers.** The first layer (the "feature transformer") turns those ~30 active
inputs into a 256-number summary called the **accumulator**, one per side. The two
accumulators (512 numbers) feed two small 32-wide hidden layers, then a final
layer outputs one number: the score in centipawns (hundredths of a pawn). The
activation between layers is a **clipped ReLU** (values squashed into a fixed
range), which matters for the integer math in [§7](#7-the-efficiency-tricks-explained-this-is-the-good-part).

Both evaluators implement the same `Evaluator` interface, so either one drops into
the search unchanged.

## 7. The efficiency tricks, explained (this is the good part)

A naive neural-net eval would be far too slow to call millions of times per move.
Three tricks make it fast. All three are in
[`engine/nn/nnue.cpp`](engine/nn/nnue.cpp), and you can measure them with
`scripts/nnue_bench` (see [§10](#10-tools-you-can-run)).

### Trick 1 — the incrementally-updated accumulator

**The problem.** Computing the 256-number accumulator from scratch means adding up
~30 columns of 256 numbers, for both sides, *every single position* the search
looks at. That's the bulk of the cost.

**The insight.** Between a position and the next one, almost nothing changes. A
move typically moves one piece (maybe captures one). So instead of rebuilding the
accumulator, **update only what changed**: subtract the columns for features that
turned off, add the columns for features that turned on. For a quiet move that's
1–2 columns instead of 30.

**How it's done here.** Each search node keeps an accumulator on a stack. On
`make()` we compute the new list of active features and apply the *difference*
versus the previous one (an O(n) merge of two short sorted lists). On `unmake()`
we just pop the stack.

**The king-move special case.** Remember that *every* HalfKP feature includes "my
king's square." So if your king moves, **every one of your features changes** —
there's nothing to reuse, and an incremental update would be slower than starting
over. So the code detects "did this side's king move?" and, only then, does a full
rebuild for that side. Every other move takes the cheap incremental path.

> This is verified bit-for-bit: `scripts/nnue_bench` plays a sequence of moves and
> checks the incrementally-updated accumulator **exactly equals** a from-scratch
> rebuild after every move, including captures, castling, promotions, and king
> moves.

### Trick 2 — quantization (integers instead of floats)

**The problem.** The net is trained in floating-point. Float math is slower than
integer math and can't use the widest SIMD lanes.

**The insight.** You don't need full float precision to play good chess. So we
**quantize**: convert the weights to small integers once, at load time, and do all
the inference in integer arithmetic. Activations are scaled by `QA = 127` (so they
fit in an 8-bit-ish range), weights by `QW = 64`. The feature weights are 16-bit
integers; the dense-layer weights are 8- or 16-bit. The final integer result is
scaled back down to centipawns.

**File formats.** The exporter can write three formats and the engine
auto-detects which it's loading:

| format | what it is                        | size     |
|--------|-----------------------------------|----------|
| NNU1   | raw float (no quantization)       | ~42 MB   |
| NNU2   | int8 weights (the default)        | ~10.5 MB |
| NNU3   | int16 feature transformer         | ~21 MB   |

The loader also accepts an older int8 layout of NNU2 for backward compatibility,
so nets exported by earlier versions still load.

> Verified faithful: the float, int8, and int16 evaluations agree to within
> rounding — quantization doesn't change how the net plays, it just makes it fast.

### Trick 3 — SIMD (do 8 multiplications at once)

**The problem.** Even as integers, the net does a lot of multiply-and-add. The
hidden layers alone are thousands of multiply-accumulates per evaluation.

**The insight.** WebAssembly has **SIMD** ("Single Instruction, Multiple Data")
instructions that operate on 128-bit vectors — eight 16-bit integers at a time.
Doing 8 lanes per instruction is roughly an 8× speedup on the inner loops.

**How it's done here:**

- The accumulator add/subtract (Trick 1) uses `i16x8` vector add/sub — 8 numbers
  per instruction.
- The dense layers — the real hotspot — use `i32x4_dot_i16x8`, a single
  instruction that multiplies eight 16-bit pairs and sums them into the running
  total. The L1, L2, and output layers are all expressed as these vectorized dot
  products. (Earlier this was the slow part because only the accumulator was
  vectorized; now the heavy layers are too.)

The same code falls back to plain scalar loops when SIMD isn't available (e.g. the
native tournament/benchmark builds), so it's always correct, just slower without
SIMD. The browser build is compiled with `-msimd128`, so the WASM contains real
SIMD instructions.

**Net effect of all three:** the per-evaluation cost dropped by roughly 3–4× and
the accumulator update went from "barely faster than a rebuild" to genuinely
incremental, which is what makes the deeper NNUE models usable in real time.

## 8. The opening book (Polyglot)

For well-known opening positions, thinking is pointless — theory already knows the
good moves. So the AI consults an opening book first.

The book is a standard **Polyglot `.bin`** file
([`frontend/book/Book.bin`](frontend/book/Book.bin), ~380k positions). Polyglot
books map a position's hash → a list of moves with weights. The probe is done in
plain JavaScript ([`frontend/js/book.js`](frontend/js/book.js)):

1. Hash the current position with the **781 standard Polyglot Zobrist constants**
   (in `frontend/js/polyglot-keys.js`). This hash is a fixed, published standard —
   it has to match the numbers the book was built with, so the constants were
   taken verbatim from the reference implementation and **verified** against it.
2. Binary-search the book for that exact hash.
3. If found, pick one of the listed moves at random, weighted by how good/common
   each is (so the AI varies its openings instead of always playing the same
   line).
4. Decode the move (Polyglot stores castling as "king onto its own rook," which is
   translated to the normal king move), and play it.

The book applies to **both** NNUE and HCE models. The moment the position isn't in
the book, the engine switches to searching as normal.

> Names of openings are **not** stored in a Polyglot file — it only holds
> moves — so book moves are simply labelled `Book move`. (Adding ECO opening names
> would require a separate dataset.)

## 9. Training the neural net

Everything in `training/` (PyTorch). The short version of how a `net.nnue` is made:

1. **Get data** (`load.py`) — download labelled positions (Lichess / Stockfish
   evaluations) as Parquet shards.
2. **Preprocess** (`preprocess.py`) — turn each position into its HalfKP features +
   target score, packed into a fast binary file (`train.bin`).
3. **Train** (`train.py`, model in `model.py` / `halfkp.py`) — gradient-descent the
   network to predict the target scores. A Colab notebook (`chess_nnue_colab.ipynb`)
   is included for training on a free GPU.
4. **Export** (`export_weights.py`) — quantize the trained weights and write
   `frontend/models/net.nnue` in one of the formats from
   [§7](#7-the-efficiency-tricks-explained-this-is-the-good-part):

   ```bash
   python training/export_weights.py --model training/checkpoints/best.pt \
          --out frontend/models/net.nnue        # default NNU2 (int8, ~10.5 MB)
   #   --float  -> NNU1 (raw float)   |   --ft16 -> NNU3 (int16 feature transformer)
   ```

Static hosts have no "upload" button, so adding a model = committing the file.
`frontend/models/manifest.json` lists which file to load.

## 10. Tools you can run

- **`scripts/nnue_bench.cpp`** — loads a net and (1) proves the incremental
  accumulator exactly equals a full rebuild, (2) times the incremental update vs a
  rebuild, (3) measures raw evaluations/second. This is how you verify the
  efficiency tricks. Build & run:
  ```bash
  g++ -O3 -std=c++17 scripts/nnue_bench.cpp engine/board/*.cpp engine/movegen/*.cpp \
      engine/rules/*.cpp engine/nn/nnue.cpp -o /tmp/nnue_bench
  /tmp/nnue_bench frontend/models/net.nnue
  ```
- **`tools/tournament`** — play two models against each other over N games and
  report win/draw/loss %. Great for "did my new net actually get stronger?"
  ```bash
  bash tools/build_tournament.sh          # Linux/Mac  (.\tools\build_tournament.ps1 on Windows)
  ./tools/tournament --a nnue:10 --b hce:10 --games 50 --net frontend/models/net.nnue
  ```
  `--a`/`--b` take `nnue:<depth>` or `hce:<depth>`. It alternates colours and
  randomises openings.
- **`tools/checknet`** — prints a net's evaluation on a few obvious positions and
  gives a quick sanity verdict (a queen should read around +9).

## 11. Building the engine (WASM)

The engine is compiled with the **Zig**-bundled clang/lld toolchain (it produces a
standard `wasm32` module; Emscripten was avoided because it was network-blocked
during development). One command:

```bash
pip install ziglang --break-system-packages      # provides the compiler
bash build.sh
```

`build.sh` compiles all of `engine/**` to `dist/engine.wasm` with `-O3 -msimd128`,
then runs `scripts/embed-wasm.js` to base64-embed it into
`frontend/js/engine-wasm.js`. After building, just reload the page.

## 13. Tweaking things

- **Opponents (the AI menu):** edit the `MODELS` object at the top of
  `frontend/js/controls.js`. Add a line like
  `AI7: { label: 'NNUE — depth 15', eval: 'nnue', depth: 15, timeMs: 8000, desc: '...' }`
  and it appears in the dropdown automatically. At high depths give a generous
  `timeMs`, or the time budget will cut the search off before it reaches the depth
  (you'll see a lower `D__` in the move readout — that's expected, not a bug).
- **Colours / theme:** the palette is the CSS variables at the top of
  `frontend/css/styles.css` (`--gold`, `--ink`, `--light-sq`, `--dark-sq`, …). The
  piece colours are two lines in `frontend/js/board-ui.js`.
- **Sounds / music:** drop-in replace the `.mp3` files in `frontend/assets/` (same
  names). The filename→event mapping is the `FILES` table in
  `frontend/js/sound.js`; background-music volume is set there too.
- **Draw-avoidance strength:** the `CONTEMPT` constant in
  `engine/search/search.cpp` (rebuild the WASM after changing it).
- **Swap in a stronger net:** export a new `frontend/models/net.nnue` (see
  [§9](#9-training-the-neural-net)) and commit it.

## 14. Honest status & limitations

- **The net is undertrained.** It plays real chess and has the right sense of
  material up to about a rook, but its evaluation saturates around +5 pawns (a
  queen reads about the same as a rook), so at equal search depth it's still a bit
  behind the hand-crafted eval. The *engine* and the *quantization* are verified
  faithful — the gap is **training** (more data, more epochs, a GPU), not code. So
  the fix is to train a better net and drop it in; nothing in the engine needs to
  change.
- **No opening names**, only book moves (a Polyglot `.bin` has no names).
- **Single-threaded search** (one worker). Fine for this scale.

The deliberate deviations from the original plan: built with **Zig** instead of
Emscripten, **inline-SVG** pieces instead of an image set, and **static hosting**
instead of any backend.
```
